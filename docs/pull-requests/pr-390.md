The preflight check suppresses all output from `gh workflow view ...` (redirecting stdout/stderr to `/dev/null`) and then reports the workflow as “not resolvable”. This can hide actionable failures (e.g., auth/permission errors, API throttling) and make the resulting error message misleading. Consider capturing and printing the actual `gh` error on failure (or only treating a 404/not-found as contract drift) so logs clearly indicate the root cause.

`REQUIRED_WORKFLOWS` is iterated via word-splitting (`for workflow_file in ${REQUIRED_WORKFLOWS}`), which is brittle if the list ever changes (and can trigger shellcheck warnings). Using a bash array (or a newline-delimited list with a `while read` loop) keeps the iteration robust and avoids accidental globbing/splitting issues.
```suggestion

          for workflow_file in "${REQUIRED_WORKFLOWS[@]}"; do
```
