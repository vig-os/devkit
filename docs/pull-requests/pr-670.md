# [PR 670](https://github.com/vig-os/devkit/pull/670) feat(nix): migrate the devcontainer to a Nix toolchain + Claude-native setup (#625)

## Migrate the devcontainer to a Nix toolchain

This epic replaces the Debian-based devcontainer (a `Containerfile` on
`python:3.14-slim`, provisioned with `apt` + ad-hoc installers) with a **Nix
flake that is the single source of truth for one toolchain**, delivered two
ways. It is a large change; this description is the reviewer's map of the new
paradigm. Per-change detail is in the `## Unreleased` section of `CHANGELOG.md`;
onboarding/architecture is in `docs/NIX.md`; the consumer-facing contract is in
`docs/MIGRATION.md` (also shipped in the image).

### The paradigm

- **One toolchain SSoT.** `flake.nix`'s `devTools` list defines the toolchain
  once. A parity test (`tests/test_flake_devshell.py`) guarantees the dev-shell
  and the image never drift. The old per-OS `requirements.yaml` dual-manifest is
  gone.
- **Two delivery modes, same toolchain.**
  - *Container image* — assembled by `dockerTools.buildLayeredImage` (no base
    distro, no `Dockerfile FROM`), **bit-reproducible**, built **natively for
    amd64 + arm64** (no QEMU). Published as `ghcr.io/vig-os/devcontainer`.
  - *Bare-nix dev-shell* — the same toolchain via `nix develop` / `direnv`,
    consumed as a flake input (`vigos.lib.mkProjectShell`). `install.sh
    --mode devcontainer|direnv|both` scaffolds either.
- **Reproducibility & updates.** `flake.lock` is the controlling version
  document; `nixpkgs` is pinned (`nixos-26.05`) with a thin `nixpkgs-unstable`
  overlay for fast-movers (`uv`, `gh`, `claude`). Renovate's `nix` manager bumps
  the lock through the normal PR/CI gate.
- **On-demand, not bloated.** The image stays minimal (build automation, git/gh,
  uv/Python 3.14, Node, shell tooling, linters, agent toolkit). Extra language
  toolchains (Rust/Go/cc) are sourced on-demand via a project flake's
  `extraPackages` or ad-hoc `nix shell`, never baked into every consumer.
- **Security.** Debian/Trivy-OS scanning is replaced by `vulnix` (nixpkgs-native
  CVE scanner) + a CycloneDX SBOM; "advance the pinned `nixpkgs` rev" is the
  primary CVE lever. A nightly `vulnix-gate` is the publish go/no-go signal.

### What reviewers should focus on

- `flake.nix` — `devTools`, `mkProjectShell`, and the `buildLayeredImage`
  assembly (the bootstrap layer that restores the FHS conveniences a pure-Nix
  userland lacks: `/usr/bin/env`, an arch-aware manylinux loader + `LD_LIBRARY_PATH`,
  a writable npm prefix, the baked project `.venv`, `/etc/nix/nix.conf`, and a
  `docker → podman` shim).
- CI: `setup-env` provisions flake tooling from the dev-shell (the SSoT) for
  every job that actually consumes it — the image build/test jobs and the vulnix
  CVE gate — while jobs that only run `uv`-managed project tools or GitHub-native
  actions use the lighter ad-hoc setup; the `Nix Image (discovery)` workflow
  builds + tests the image on both arches and runs an in-container Nix runtime
  smoke test.
- The dev-shell ⇄ image parity model and the NixOS-vs-FHS handling of the C++
  runtime on `LD_LIBRARY_PATH` (gated to avoid leaking into FHS host binaries).

### Consumer-facing contract (see `docs/MIGRATION.md`)

No `apt`; `#!/usr/bin/env` works; `npm -g` resolves; `docker` maps to `podman`
(DooD via mounted socket); Python is uv-managed CPython 3.14 (pin
`requires-python` as a range, not an exact patch); manylinux wheels run;
pre-commit `ruff`/`typos` come from the flake.

### Cross-stack validation (final image)

The finished image + bare-nix dev-shell were exercised live against one real
consumer per stack, in **both** delivery modes (container `:dev` and
`nix develop` against the flake input):

| Repo | Stack | What was tested | Result |
|------|-------|-----------------|--------|
| `vig-os/sync-issues-action` | TypeScript | post-create (node/`act`/`tsx`/`local-action`), `npm ci && build && test`; bare-nix `python3`+`pre-commit` | ✅ 95/95 |
| `exoma-ch/brother-printer` | Python | baked `.venv` + post-create, `uv sync`, in-image pre-commit incl. `pymarkdown`, `just test`, DooD | ✅ 321 pass |
| `exo-pet/playground-carlos` | Python/web | post-create, numpy/scipy/pandas + `pymarkdown`/`ruff`/`typos` in-image, `just test` | ✅ 69 pass |
| `vig-os/devcontainer-smoke-test` | Shell + DooD | `docker`→`podman` DooD (sibling container spawn), on-demand `nix` | ✅ (see note) |
| `vig-os/scitadel` | Rust | image stays Rust-minimal; on-demand toolchain via project-flake `extraPackages` / `nix shell` | ✅ |

**What a consumer must adapt for the new environment** (the image is a pure-Nix
userland, not Debian):

- **Drop `apt`-based `post-create` steps** — there is no `apt`. Get tools from the
  baked toolchain, `uv`/`npm`, or Nix. (e.g. the old node-via-`apt` pattern and
  brother-printer's `apt install libusb` step now no-op; the latter only affects
  opt-in hardware tests.)
- **Pin `requires-python` as a range**, not an exact patch — an `==3.14.x` pin can
  be unsatisfiable against the image interpreter. (The smoke-test repo still
  carries a stale `==3.14.6`; it heals from the template at the next release
  dispatch — the ⚠️ "note" above.)
- **Compiled toolchains** (Rust/Go/cc) come from the consumer's **project flake**
  (`extraPackages`) or an ad-hoc `nix shell`, never the base image.
- **DooD** needs both `CONTAINER_HOST` and `DOCKER_HOST` pointed at the mounted
  socket (the scaffolded `docker-compose.yml` does this); literal `docker` calls
  resolve through the `podman` shim.

### Scope / sequencing

This PR is validated end-to-end (CI green on both arches; a cross-stack live
sweep across TypeScript/Python/Rust/shell consumers in both modes). The
versioned/`:latest` **publish-cutover is intentionally gated** and remains a
separate, deliberate step after this merges.

### Naming follow-up: the repo is no longer "a devcontainer"

Not a blocker for this PR — flagging for a **separate, post-cutover** follow-up.

After this migration the repo is **a Nix flake that is the single source of truth for one toolchain, delivered two ways** — an OCI image *and* a bare-nix dev-shell (`mkProjectShell`) — plus the agent/Claude scaffolding. The name `devcontainer` now describes only one of the two delivery modes and still implies the deleted Debian/`Containerfile` paradigm. `flake.nix`'s own description already reframes it: *"toolchain SSoT (dev-shell + image basis)."*

**Suggested name: `vig-os/devkit`** — broad enough to cover both the image and the dev-shell, reads as "the kit every repo pulls in," and avoids the `devenv.sh` name-clash that `devenv` would create in Nix circles. Runners-up considered: `toolchain` (names the architecture directly) and `workbench`.

This is more than a label, so it should be sequenced **after** the gated publish-cutover, not folded into this PR:

- The image path is hardcoded as `ghcr.io/vig-os/devcontainer` (`flake.nix:599`, mirrored in `nix-image.yml` + the testinfra targets). Renaming the GHCR path is a consumer-breaking change on its own and should ride with (or after) the publish cutover.
- A rename sweep would also touch `org.opencontainers.image.source` (`flake.nix:654`), `CODEOWNERS`, and doc cross-links. GitHub auto-redirects the old repo URL, but the baked OCI label and image ref won't redirect.

Proposing we capture this as its own issue rather than expanding the scope here.

## [Comment #2](https://github.com/vig-os/devkit/pull/670#issuecomment-4834503213) by [@gerchowl](https://github.com/gerchowl)

### Review — reconciled findings (CI · flake · scripts · security · tests · docs)

Thanks @c-vigo — this is genuinely strong work: the flake is meticulously documented, every FHS-gap workaround carries an issue ref, and the SSoT framing is sound. Below is a consolidated technical review from a multi-area pass. A separate roadmap comment covers the forward-looking/out-of-scope items, and the [rename note](https://github.com/vig-os/devcontainer/pull/670#issuecomment-4834414012) is already posted.

**Framing:** the versioned publish is intentionally gated, so the two "blockers" below don't block *this* merge — they're **must-fix before the separate publish-cutover step**.

#### Blockers (before the publish-cutover)

- **B1 — `release.yml` retag will fail.** The flake hardcodes `tag = "nix-dev"` (`flake.nix:603`); `build-image/action.yml` saves the tar with that tag and never retags. But `release.yml:954-956` does `docker tag "$REPO:$BUILD_VERSION-$arch" …` → "No such image", so publish never pushes. Fix: capture the loaded tag dynamically (`nix-image.yml:152` already does this via `sed -n 's/^Loaded image: //p'`) and retag from that. *(Verified.)*
- **B2 — "vulnix-gate is the publish go/no-go" isn't wired up.** `vulnix-gate` lives only in the nightly `security-scan.yml`; `release.yml` has no `needs:`/required-check linking publish to it. The only release-path CVE gate today is Trivy, which the repo itself documents as "largely dark" on a Nix image. Fix: run `vulnix-gate` as a hard gate in the release path, or make the nightly a required pre-publish check. *(Verified.)*

#### Major

- **M1 — the "never drift" parity claim is overstated.** `tests/test_flake_devshell.py` only reads `devShellTools` and runs each binary in `nix develop` — it **never touches the image side**, and it's one-way (removing a tool from `devTools` keeps it green). Image coverage in `test_image.py` is hand-curated and misses ~half of `devTools` (`lazygit`, `delta`, `jq`, `shellcheck`, `rg`, `fd`, `bat`, `eza`, `nvim`, `claude`, …) plus most of the FHS substrate. Fix: parametrize a presence-on-PATH test over `nix eval .#devShellTools` **inside the container** — turns the SSoT from a half-promise into an actual gate.
- **M2 — `vulnix_gate.py` silently passes in failure modes.** Unscored CVEs (`score is None → continue`, `vulnix_gate.py:69`) are treated as LOW — a CVE disclosed yesterday with no NVD score won't block. And `security-scan.yml:82` runs vulnix with `|| true`, so a scanner crash emitting `[]` is indistinguishable from a clean scan. Both want fail-loud handling.
- **M3 — "setup-env provisions every job from the flake" is only partly true.** python-security, security-scan, dependency-review, and release validate/publish call it *without* `provision-via-flake: 'true'`. Either make it consistent or scope the claim in the PR body.
- **M4 — script bugs.** `install.sh:520` `--skip-pull` uses `$RUNTIME image exists`, which is podman-only → breaks docker users (use `image inspect`). `ci.yml` python-security runs `safety check` without `uv run` behind `|| true`, so it can silently never run (and `safety check` is deprecated in 3.x).
- **M5 — two docs lag the shipped code.** `docs/NIX.md` (publish-cutover section) and `docs/CONTAINER_SECURITY.md` still describe the vulnix gate as "non-blocking / discovery phase" and reference a `builder: debian|nix` selector that no longer exists. Re-sync before merge.

#### Minor / nits

- `release.yml` summary treats a `cancelled` required job as green.
- `nix-image.yml` branch filter strands arm64 image testing once this merges (`ci.yml` build-image is amd64-only).
- `scripts/init.sh` "Pre-commit hooks installed" is misleading (`install-hooks` only fetches envs).
- Unparenthesized `except A, B:` tuples in a few tests (`test_claude_ssot.py:53,74`, `conftest.py`) — works on 3.14, fragile.
- Network-dependent image tests (`uv add`, `npm i -g tsx`) have no offline skip-guard.
- `.vulnixignore` Class-2 uses a blanket rationale over 22 HIGH/CRITICAL CVEs (incl. CRITICAL openssl) — wants per-CVE notes / shorter expiry.
- `install.sh` dry-run output is hand-maintained beside the real `CMD=(...)` (use `printf '%q'`); `git add -A && commit "initial scaffold"` can sweep a populated dir.
- `version-check.sh` help text typo: `"And (days)"` → `"Nd (days)"`.

#### Credit where due

- The `assets/workspace/**` duplication is **not** a DRY violation — it's hook-enforced scaffolding (`scripts/sync_manifest.py` + `sync-manifest` pre-commit hook + `test_claude_ssot.py`), byte-identical except one declared transform.
- `Containerfile` cleanly removed (remaining mentions are test fixtures + historical changelog).
- Actions are SHA-pinned with version comments; release uses GH App tokens (the good pattern); image builds natively per-arch (no QEMU).

## [Comment #3](https://github.com/vig-os/devkit/pull/670#issuecomment-4834503378) by [@gerchowl](https://github.com/gerchowl)

_Posted on June 29, 2026 at 03:58 PM_

### Forward-looking roadmap — follow-ups (NOT in scope for #670)

Capturing the architecture threads we discussed so they don't get lost. None of these should expand this PR — they're proposed as separate, sequenced issues. Flagging the same intellectual honesty throughout: where a claim is "relocate" not "eliminate," I say so.

#### A. Nix-world gaps (highest leverage)

- **Supply-chain trapdoor:** `accept-flake-config = true` is baked into the image's `/etc/nix/nix.conf` (`flake.nix:591`). Any in-container `nix run github:attacker/flake` silently accepts that flake's `substituters`/`trusted-public-keys` — the textbook Nix cache-redirection attack. Replace with explicit `substituters` + `trusted-public-keys`; require `--accept-flake-config` per-invocation.
- **Silent drift:** `uvPythonDownloadsJsonUrl` (`flake.nix:146`) is pinned to literal uv `0.11.23` in a raw-githubusercontent URL while `uv` floats from `nixpkgs-unstable`, and Renovate won't touch it. Derive the URL from `pkgs.uv.version`; add a Renovate regex-manager rule for the `pip-licenses` wheel hash (`flake.nix:276-281`) too.
- **The FHS/manylinux/loader scar tissue** (`#683/#697/#698/#703/#728/#735/#736/#740`) is one root cause — uv's generic-ELF wheels vs no-FHS Nix. `uv2nix` / `pyproject-nix` would dissolve most of it and make `pythonEnv` the real hermetic venv. Worth a scoping issue even if not adopted now.
- **Cachix is opportunistic** (`continue-on-error`, paths-filtered, only pushes the dev-shell *runtime* closure — not the image), so consumers may rebuild the whole closure from source. Make the image-closure push first-class and blocking.
- **`eachDefaultSystem` advertises the Linux-only image on darwin** (breaks `nix flake check --all-systems` on a Mac), and `import nixpkgs-unstable` lives inside the overlay closure so it re-evaluates per attribute access. Both cheap fixes.
- Nice-to-haves: `treefmt-nix`, `deadnix`/`statix` checks, `nixosModules`/`homeManagerModules` outputs, a `nix run .#install` app, and `nixfmt --check` globbing all `*.nix` (today only `flake.nix`).

#### B. git-hooks.nix + prek (supersedes #40)

prek and git-hooks.nix aren't competitors — different layers. Adopt **git-hooks.nix with `package = pkgs.prek`**: generates `.pre-commit-config.yaml` from Nix (no drift from `devTools`), drops the Python `pre-commit` from `imageTools` (one fewer manylinux/FHS consumer), and exposes `checks.pre-commit` so the hooks run under `nix flake check`. This lands **#40** and the flake-SSoT wiring in one move. ⚠️ prek's YAML parser is stricter than pre-commit's and may reject hooks pre-commit tolerated — needs a CI shakeout.

#### C. Flake-checks-as-CI — Tier 0 (stay on GitHub, kill the YAML sprawl)

The boundary is hard: checks are sandboxed (no network, no recursive Nix, no live containers, no secrets, host-arch only) — `flake.nix:370-379` already documents why the parity test can't be a check. Push *more* across that line:

| Moves into `checks.<system>.*` (pure) | Stays GH-orchestrated (effectful) |
|---|---|
| nixfmt, ruff, typos, shellcheck, hadolint, taplo, deadnix/statix | Image testinfra (`test_image.py`) — live `podman run` |
| Pure pytest units (`test_utils`, `test_vulnix_gate`, `test_claude_ssot`, `check-expirations`) | Integration — network (`uv sync`, `npm ci`, manylinux PyPI) |
| dev-shell build, `devShellTools` eval *(already)* | Parity test — recursive `nix`; vulnix — CVE-DB network |
| `checks.pre-commit` (from B) | Multi-arch matrix; publish/cosign/SBOM/cachix push |

Drive it with a Nix-native runner instead of bare `nix flake check` (monolithic, poor caching): **garnix** (hosted GH App, *zero* workflow YAML) or **`nix-fast-build` / `om ci`** on existing runners (no new vendor, parallel + cachix-aware). The effectful jobs stay as-is.

**Tier 1 (mentioned, future escalation only):** if the CI *engine* should later move off GitHub-hosted runners while keeping the forge — self-host via **Hydra** (canonical flake-evaluator CI) or **Buildkite** (self-hosted runners, hosted UI). Separate decision.

#### D. Secrets — sops-nix/age + OIDC

For a toolchain-SSoT repo, ship the secret-management *pattern*, not just the toolchain: **sops-nix + age** for runtime + downstream-consumer secrets (decrypt with the consumer's own key, no per-repo GH secret dance), and **OIDC** for cloud/registry auth. Honest caveat: on GH-hosted runners SOPS **relocates** the root of trust to one bootstrap key — it doesn't eliminate the GH secret. OIDC is the real "no stored secret" lever; keep the existing GH App installation-token model (it's the good pattern). 

#### E. Rename → `devkit`

See the [rename comment](https://github.com/vig-os/devcontainer/pull/670#issuecomment-4834414012) — sequence it after the gated publish-cutover (the GHCR path + OCI `source` label are consumer-facing).
