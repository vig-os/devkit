# [PR 1033](https://github.com/vig-os/devkit/pull/1033) fix(ci): make scaffolded CI and release pipeline language-neutral

## Summary

Bundles the two pilot-surfaced "CI layer is Python-shaped" bugs (both from vig-os/commit-action#32) — same theme, one CI run.

### #1028 — gate python/uv env in `setup-devkit-toolchain`
All three Python-assuming spots now gate on `pyproject.toml` at the repo root: `UV_PROJECT_ENVIRONMENT` (container branch), the Nix-CPython `$GITHUB_PATH` filter, and the `UV_PYTHON_DOWNLOADS_JSON_URL` forward (direnv branch). On a non-Python consumer the composite is a no-op for those steps — and the Nix python stays on PATH, removing the latent footgun. `DEVKIT_MODULES` was verified still Reserved/empty in `.vig-os`, so file presence is the gating signal (the issue's primary suggestion). The devkit itself has a root `pyproject.toml`, so its own CI is unchanged.

### #1029 — language-neutral release pipeline with opt-in bundle
- Neutral labels: "Sync Python dependencies" → "Sync dependencies" (`release-core.yml`), "Pytest" → "Run tests" (`ci.yml` header). Remaining `python` hits in the scaffold workflows are genuinely Python-specific and untouched.
- Opt-in release artifact: the `finalize` job detects a `bundle` just recipe (`just --summary | grep -qw bundle`), runs `just bundle`, and adds `dist` to the existing finalization commit's `FILE_PATHS` — so the tagged commit ships a fresh bundle. Zero new config surface: the recipe's presence is the flag, matching the capability-by-presence idiom of the #1028 fix. Verified `commit-action`'s `FILE_PATHS` supports directories ("directory expansion" per its README) and that `commit-action` itself already defines a `bundle` recipe.
- Release choreography untouched (epic #988 keeps it single-source) — the step is additive and gated.

TDD: each fix lands as failing-test commit → implementation commit (`tests/bats/init-workspace.bats`, 3 new shape tests).
