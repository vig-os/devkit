# [PR 1190](https://github.com/vig-os/devkit/pull/1190) fix(workspace): isolate dev-shell env dump from shellHook stdout

## Summary

1.4.0-rc2 consumer validation failed on every direnv-mode host CI job (org-config vig-os/org-config#54, sync-issues-action vig-os/sync-issues-action#143) at "Set up devkit toolchain":

```
##[error]Unable to process file command 'env' successfully.
##[error]Invalid format 'devcontainer dev environment loaded (nix)'
```

The #1180 shellHook env forward captured the **whole stdout of `nix develop`** as the env dump, so the scaffolded flake's shellHook banner glued onto the first NUL-delimited record and produced an invalid GITHUB_ENV heredoc name. Both banner and action are devkit-managed — every direnv consumer breaks and cannot fix it locally. rc2's smoke test missed it because the smoke-test consumer is container-mode.

## Fix (TDD)

- `dcbcbd02` **test (RED)**: stub `nix` now echoes the banner on stdout on every invocation, exactly like the real flake; asserts the banner never reaches GITHUB_ENV and the first env record survives. Reproduced the exact live corruption.
- `20b6d58b` **fix (GREEN)**: dump the env to a file *inside* the shell command (`--command bash -c 'env -0 > "$1"' _ <file>`) so shellHook stdout stays on the job log; plus defense in depth — skip any record whose name is not a valid env identifier. 19/19 tests green.

The step's other two stdout captures (PATH, UV download URL) are already banner-safe via their `grep` filters; the env dump was the only raw capture.

Note: 4 unrelated local pytest failures against the stale local `:dev` image (missing DOWNSTREAM_RELEASE.md / 1.4.0 devtools) — CI builds fresh.

Per the RC-validation runbook this targets `release/1.4.0`; rc3 follows. Second rc finding after #1187.
