# [PR 998](https://github.com/vig-os/devkit/pull/998) chore(ci): spike conditional container semantics (#992)

## Spike #992 — conditional job-level `container:` semantics

Validates the mode-branching mechanism for the workflow refactor (#991),
deciding **Option A vs B** for epic #988.

**Question:** can a single workflow conditionally run a job in a container OR
on the host runner, driven by a preceding job's output — including across a
`workflow_call` boundary?

The two files under `.github/workflows/spike-992-*.yml` are **temporary**;
they run on `pull_request` (no branch filter) purely to exercise these cases on
real GitHub runners and will be **removed** before this branch merges. The
persistent artifact of this spike is the ADR and the run-log URLs.

Cases exercised (each prints `hostname` / `id -u` / `$HOME` / `/etc/os-release`):

1. empty `image` expression, no credentials → host or error?
2. empty `image` expression, WITH `credentials:` → host or error?
3. non-empty image (`ubuntu:24.04`) → container baseline
4. a/b: same pattern inside a reusable `workflow_call` workflow (image threaded via inputs)
5. whole `container:` value as an expression yielding a map-or-`null`

Some cases may FAIL by design — a failure that proves "empty image errors" is a
valid spike result and its exact error text is captured in the ADR.

Refs: #992
