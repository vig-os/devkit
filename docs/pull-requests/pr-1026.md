# [PR 1026](https://github.com/vig-os/devkit/pull/1026) fix(ci): make commit-message enforcement real

## Description

Closes the stale commit-scope allowlist from #1019 — and the two deeper defects behind it.

The scope list was a *symptom*. The root cause is that **both enforcement paths were dead**, so the config was free to rot for hundreds of commits:

1. **The scope allowlist was stale** — the five-scope pin (`agent,ci,setup,image,vigutils`) rejected **594 of the 1206 scoped commits in history (~49%)**, and contradicted `docs/COMMIT_MESSAGE_STANDARD.md`, which already defines a scope as free-form *"alphanumeric and hyphens only"*.
2. **CI never validated commit messages.** `validate-commit-msg` is a `commit-msg`-stage hook, and `prek run --all-files` (the `project-checks` lane) does not run stage-gated hooks. The docs claimed CI enforcement in three places; it never happened.
3. **Scaffolded repos had no commit validation at all.** They shipped a `.githooks/commit-msg` shim and a `COMMIT_MESSAGE_STANDARD.md`, but their rendered pre-commit config had **zero** `commit-msg`-stage hooks — so the shim was a no-op in every consumer repo.

### ⚠️ The blocker that was not in the issue

Turning CI enforcement on as-configured would have **failed every Renovate/Dependabot PR**. Those bots emit `build(pip): lock file maintenance` and `ci(actions): update …` with **no `Refs:` line**, and `--refs-optional-types` only covers `chore`. Measured on a recent 150-commit window: **18 of the 94 bot commits would have failed.**

The docs claimed *"the commit-msg hook and CI validator enforce these exemptions automatically"* — there was no author-based exemption in the validator at all. This PR adds one, keyed on the `[bot]` author suffix, and pins it with tests. The same message from a human is still rejected.

## Changes Made

**Scope vocabulary is now free-form.** Dropped `--scopes` from `nix/hooks.nix` (the SSoT — both `.pre-commit-config.yaml` renders derive from it). This required **zero validator code**: with `--scopes` omitted, scope enforcement is off while the subject regex still enforces `[a-zA-Z0-9-]+` — precisely the documented rule. Type, `Refs:`, and the agent blocklist remain enforced. `gh_issues` (underscore) is still rejected.

**New `validate-commit-range` entry point** (`packages/vig-utils/`, 33 tests). Reuses the existing `validate_commit_message()`; walks a git range, skips merge commits and bot-authored commits, and validates a PR title. Putting the logic in Python rather than a shell loop in YAML is what makes it unit-testable and reusable by the scaffolded consumer CI.

**New `commit-checks` CI job** in devkit's *and* the scaffold's `ci.yml`. It validates every commit the PR adds (from the merge-base with the base branch — not `base.sha`, which would drag already-landed commits into the range if the branch merged `dev` back in) plus the **PR title**, which becomes the merge commit's subject under `--no-ff`. The PR title is passed via `env:`, never interpolated into the `run:` block.

**Scaffold gap closed.** `validate-commit-msg` and `prepare-commit-msg-strip-trailers` now reach the consumer render. They ship together on purpose: the prepare hook strips agent trailers *before* the validator's blocklist gate sees them, so an agent-authored commit is repaired rather than hard-rejected in a consumer repo.

**Docs corrected** to describe what is actually enforced: free-form scopes, the bot/merge-commit exemptions, and the local-hook vs `commit-checks` split.

Not in scope, per the issue: **existing history is not rewritten.**

## Changelog Entry

### Added

- **Commit messages are validated in CI** ([#1019](https://github.com/vig-os/devkit/issues/1019))
  - New `commit-checks` job (devkit and scaffolded repos) runs `validate-commit-range` over every commit a pull request adds, plus the **pull request title** — which becomes the merge commit's subject under `--no-ff`. `validate-commit-msg` is a `commit-msg`-stage hook, so `prek run --all-files` never ran it: until now the standard was enforced only by a local hook, and only on a machine whose `core.hooksPath` was intact.
  - Merge commits and bot-authored commits (`…[bot]`) are exempt. Renovate and Dependabot emit `build(pip): …` / `ci(actions): …` with no `Refs:` line, so without the exemption the new gate would fail every dependency PR. The exemption is keyed on the author — the same message from a human is still rejected.
- **Scaffolded repos can enforce the commit standard** ([#1019](https://github.com/vig-os/devkit/issues/1019))
  - `validate-commit-msg` and `prepare-commit-msg-strip-trailers` now reach the consumer pre-commit config. Scaffolded repos already shipped a `.githooks/commit-msg` shim and a `COMMIT_MESSAGE_STANDARD.md`, but had no `commit-msg`-stage hooks for the shim to run — the documented standard was unenforceable in every consumer repo.

### Changed

- **Commit scopes are free-form** ([#1019](https://github.com/vig-os/devkit/issues/1019))
  - The `validate-commit-msg` hook no longer pins an allowlist of commit scopes. The previous five-scope list (`agent,ci,setup,image,vigutils`) rejected 594 of the 1206 scoped commits in history (~49%), including the scopes used by our own bots, and contradicted `docs/COMMIT_MESSAGE_STANDARD.md`, which defines a scope as free-form "alphanumeric and hyphens only". The commit **type**, the `Refs:` line and the agent blocklist remain enforced; only the scope vocabulary is open.

### Manual Testing Details

**Dogfooded the new CI lane against this very branch** (the exact `commit-checks` logic, from a fresh `nix develop`):

```
Validating 973f4364..5a8c9e9b (base: dev)
  commit-checks would PASS (exit=0)
```

**The gate is not vacuously passing.** Run against a real 150-commit slice of `dev`, it found three genuine violations:

```
Commit b91d7823: 'perf(image): bake placeholder manifest into the Nix image'
  Unknown commit type 'perf'.
Commit f566fff6: 'fix(devshell): isolate neovim from host ~/.config/nvim'
  Missing mandatory 'Refs: <IDs>' line.
Commit 08adf1f0: 'docs(changelog): consolidate renovate dependency updates …'
  Missing blank line and 'Refs: <IDs>' after the subject line.
```

**The bot exemption is load-bearing** — same range, bypassing the exemption:

```
non-merge bot commits: 94
would FAIL without the exemption: 18
  renovate[bot]   build(pip): lock file maintenance
  renovate[bot]   ci(actions): update cachix/install-nix-action action to v31.10.7
  …
```

**Scope behaviour** — previously-rejected real scopes now pass; malformed ones still fail:

| message | before | after |
|---|---|---|
| `fix(workspace): point scaffolded flake stub …` | rejected | **passes** |
| `chore(nix): bump nixpkgs-unstable` | rejected | **passes** |
| `fix(gh_issues): …` (underscore) | rejected | rejected |
| `wibble(ci): …` (unknown type) | rejected | rejected |
| `feat(ci): …` (no `Refs:`) | rejected | rejected |

**Scaffold shim is no longer a no-op** — `commit-msg`-stage hooks in the consumer render: **0 → 2**.

Two follow-ups surfaced while doing this, deliberately **not** bundled here:

1. **`perf` is not an approved commit type**, but `perf(image): …` exists in history — so the *type* vocabulary has drifted too, not just scopes. Harmless today (CI only validates a PR's own commits), but it deserves a decision: add `perf`, or leave it rejected.
2. **`check-agent-identity` is still not scaffolded** (#163). It guards the git *author/committer*, a distinct concern from message validation, so scaffolding it would have been scope creep. Worth its own issue if consumers should have it.
