# [PR 791](https://github.com/vig-os/devkit/pull/791) build: adopt git-hooks.nix + prek runner, expose checks.pre-commit, and gate flake checks as CI Tier 0

## What

Two related build-system changes land together on this branch:

1. **#778 (Closes #40) — pre-commit → prek.** Adopt `cachix/git-hooks.nix` with
   `package = pkgs.prek` and migrate the git-hook runner from the Python
   `pre-commit` to the Rust **prek**. The flake now exposes
   **`checks.pre-commit`** (the sandbox-pure hook subset, run under the flake
   check tier), `prek` joins the `devTools` SSoT (dev-shell + image), and the
   standalone `pre-commit` is dropped from both — one fewer manylinux/FHS
   consumer in the image closure.
2. **#779 — CI Tier 0 via nix-fast-build.** Build every `checks.<system>`
   derivation (treefmt formatting, deadnix/statix, dev-shell, `devShellTools`,
   and the new `checks.pre-commit` gate) in parallel with an eval cache via
   **`nix-fast-build`**, replacing the serial `nix flake check` step in
   `project-checks`. Recorded in `docs/rfcs/ADR-flake-checks-tier0.md`.

> **Scope note:** #779 depends on #778's `checks.pre-commit` existing, so the two
> are landed together rather than split. Both issues are addressed here; the
> commit trail keeps them on separate `Refs:` (`#778` / `#779`).

## Design: two hook artifacts, kept in agreement

`checks.pre-commit` builds in the Nix sandbox (no network, no venv), so it can
only run **pure** hooks. Rather than generate a store-path-bound config (which
would not be portable to the downstream scaffold and would churn on nixpkgs
bumps), the committed `.pre-commit-config.yaml` stays the hand-maintained,
PATH-based **runner** config, and `checks.pre-commit` is the Nix-verified
guarantee that its pure hooks are correct. Documented in `docs/NIX.md`.

### Hook → mechanism mapping

**In `checks.pre-commit` (pure, git-hooks.nix + prek):**
- formatting → one `treefmt` hook reusing `treefmtEval` (nixfmt + ruff-format + taplo), replacing the individual nixfmt/ruff-format/taplo-format hooks
- pure linters → `ruff`, `shellcheck`, `yamllint` (`--format parsable --strict`), `typos`, a custom `taplo-lint`, and `just --fmt --check --unstable`
- pre-commit-hooks meta hooks (built-ins) → check-added-large-files, check-case-conflicts, check-json, check-merge-conflicts, check-symlinks, check-toml, check-yaml, python-debug-statements, detect-private-keys, end-of-file-fixer, mixed-line-endings, trim-trailing-whitespace
- vig-utils/bandit → wired to hermetic Nix binaries (`${vigUtils}/bin/…`, `${pkgs.bandit}/bin/bandit`): check-action-pins, check-skill-names, check-expirations, bandit

**Runner-only (excluded from `checks.pre-commit`):** generate-docs, sync-manifest, pip-licenses, pymarkdown (not in nixpkgs), no-commit-to-branch, destroyed-symlinks (no git-hooks.nix built-in + git-state-dependent), check-agent-identity (inspects git identity), and the commit-msg/prepare-commit-msg-stage hooks.

### Fidelity notes
- `python-debug-statements` package pinned to the **3.14** `pre-commit-hooks` so it parses the repo's PEP-758 `except A, B:` (nixpkgs' default is 3.13, which rejects it) — matching the runner interpreter.
- `shellcheck` moved from the `shellcheck-py` remote hook to a `language: system` hook (flake `shellcheck`): the wheel bundles a manylinux binary the Nix userland can't exec and prek/uv can't build it under 3.14 — same reasoning as ruff/typos/nixfmt in #697/#701.
- `check-yaml` runs with `--allow-multiple-documents` in **both** the runner and the Nix check (git-hooks.nix' built-in hardcodes `--multi`), so the gate is not more lenient than the runner on multi-document YAML.

## Runner/image/scaffold changes
- `.githooks/*` (root + scaffold) call `prek run`; `scripts/init.sh` runs `prek prepare-hooks`; `just precommit` → `prek run --all-files`; worktree setup → `prek install -t pre-commit -t commit-msg -t prepare-commit-msg`; scaffold `init-precommit.sh`/`justfile.project` follow.
- Image: `prek` via devTools (no `pre-commit`), baked cache renamed `PREK_HOME=/opt/prek-cache`, `precommit` alias → `prek run`.
- CI lint gate (`.github/actions/test-project`) runs `prek run --all-files` (was `uv run pre-commit run --all-files`); `pre-commit==4.6.0` dropped from `pyproject.toml` + `uv.lock` and the vestigial `pre-commit-` dev-shell PATH exclusion removed.

## Review follow-ups (addressed on this branch)
- **check-yaml parity** — the Nix gate hardcodes `--multi`; the runner now passes `--allow-multiple-documents` too so both agree (was: gate strictly more lenient).
- **worktree hook stages** — `prek install` now installs all three stages; bare `prek install` only wired the pre-commit shim, silently dropping commit-msg / prepare-commit-msg in worktrees.
- **scaffold completeness** — repointed the lint job's dead `PRE_COMMIT_HOME=/opt/pre-commit-cache` → `PREK_HOME=/opt/prek-cache`, updated `container-ci-quirks.md`, and dropped the stale `.pre-commit-cache` idempotency guard in `init-precommit.sh`.
- **test fidelity** — `init.bats` now asserts the real `prek prepare-hooks` command instead of a string that only matched a leftover comment.

## Verification
- Tier-0 flake checks via `nix run .#nix-fast-build` → **all checks passed** (formatting, deadnix, statix, devShell, devShellTools, **pre-commit**).
- `prek run --all-files` → green on the root repo **and** the nested scaffold (all hooks incl. the impure `uv run` ones).
- `prek validate-config` → passes on the root and `assets/workspace` configs.
- `nix build .#devcontainerImage` builds; the image closure has `prek` (0.3.11) and no `pre-commit`.
- Image + flake parity tests updated (TDD, `test:` committed first); `tests/bats/init.bats` green (29/29).

Closes #40, #778, #779

## [Comment #2](https://github.com/vig-os/devkit/pull/791#issuecomment-4864288725) by [@gerchowl](https://github.com/gerchowl)

_Posted on July 2, 2026 at 09:35 AM_

**Review finding — migration gap: CI lint gate still on `pre-commit`**

`.github/actions/test-project/action.yml:85` is untouched by this PR and still runs:

```
uv run pre-commit run --all-files --show-diff-on-failure --color=always
```

This PR removes `pre-commit` from the dev-shell and image and advertises a full runner migration to `prek`, but the CI job that exercises the **whole** committed hook suite still invokes the Python `pre-commit` (`pre-commit==4.6.0` is still in `pyproject.toml` + `uv.lock`; `prek` is in neither).

Consequences:
- `prek` is never validated against the full committed config in CI — only the pure subset via `checks.pre-commit` plus the single-file container smoke test — so a `prek`-vs-`pre-commit` behavioral divergence on the impure hooks (generate-docs, sync-manifest, pip-licenses, pymarkdown, check-agent-identity, commit-msg stages) would ship undetected.
- `pre-commit` remains a project dependency, contradicting the "dropped from both" narrative.

Suggest either switching this step to `prek run --all-files` or explicitly documenting that the CI gate intentionally stays on `pre-commit`.

## [Comment #3](https://github.com/vig-os/devkit/pull/791#issuecomment-4864288851) by [@gerchowl](https://github.com/gerchowl)

_Posted on July 2, 2026 at 09:35 AM_

**Review finding — `just-fmt` silently dropped from `checks.pre-commit` and undocumented as runner-only**

The `just-fmt` hook (`.pre-commit-config.yaml:102`, `entry: just --fmt --unstable`, `language: system`) is sandbox-pure and `just` is already in `devTools`, yet it appears in neither the `preCommitCheck` hooks attrset (`flake.nix`) nor the treefmt wrapper (which covers only nix/python/toml).

It is also absent from the runner-only exclusion list enumerated in `docs/NIX.md` and the ADR — unlike `nixfmt` (folded into treefmt) and `pymarkdown`/`no-commit-to-branch`/etc. (explicitly excluded). It is the only pure hook in the committed config that is neither mirrored nor documented as excluded, so justfile-formatting drift is not caught by the Nix-verified gate this PR positions as "the guarantee that the pure hooks stay correct."

Suggest adding a `just-fmt` custom hook (mirroring the `taplo-lint` pattern) or documenting it as a deliberate exclusion.
