# [PR 1069](https://github.com/vig-os/devkit/pull/1069) fix(setup): resolve upstream and issue references in the Renovate preset

## Summary

The shared preset's `lockFileMaintenance.description` (added in #1049) is rendered by Renovate **into every consumer's Dependency Dashboard issue**, so its references are resolved in the *consumer's* repo context. Two of them were broken there:

- `renovatebot/renovate#41825` — the upstream blocker is a GitHub **Discussion**, not an issue, so the `owner/repo#N` shorthand resolves to nothing (`gh` returns 404 on it). Now the full [discussions URL](https://github.com/renovatebot/renovate/discussions/41825), described as the open idea thread it is: unanswered, *Suggest an Idea* category, 23 upvotes, last activity 2026-06-30 — not a planned item with a timeline, as the earlier wording implied.
- Trailing `Refs #1041` — a bare reference that GitHub autolinks to the **consumer's** issue 1041. Now the absolute devkit issue URL, the same treatment #1056/#1062 gave doc paths (this one differs in that it is actually *rendered*, not sitting in a YAML comment).

The `#1049` CHANGELOG entry already linked the discussion correctly, so the preset and the re-scope comment on #1041 were the only two places carrying the broken shorthand. [The comment is updated](https://github.com/vig-os/devkit/issues/1041#issuecomment-4967504453).

Targets `release/1.2.0`: the preset it fixes ships in 1.2.0, so the CHANGELOG entry lands in that section (mirror re-synced via `sync-manifest`).

**Does not close #1041.** That issue still tracks the open upstream scope — alert-driven remediation of out-of-range transitive vulns, and transitive packages appearing in the Dependency Dashboard at all.

Text-only, no behaviour change, so no test accompanies it; the preset's contract is covered by `renovate-config-validator --strict` in `renovate-validate.yml`.
