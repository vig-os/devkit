# [PR 1156](https://github.com/vig-os/devkit/pull/1156) docs(release): document first-release floating-tag ruleset bypass

## Summary

Companion to #1151. The imported **Tag ruleset bypasses only the Release App
(Integration)** — correct for steady state, where `promote-release.yml` moves
`<prefix>X` / `<prefix>X.Y` with the app token. But on a **first release** the
promote workflow is not dispatchable (#1151) and **no human — not even an
admin — can move the floating tags** (`422 Cannot update this protected ref`).
The release then publishes with `<prefix>X` stale and `<prefix>X.Y` missing,
silently breaking the advertised `uses: owner/repo@<prefix>X` pin.

## Fix (issue preference #1 — docs)

Adds a **"First-release floating tags"** subsection under the first-release
runbook in `docs/MIGRATION.md`:

1. Temporarily grant repository admins a bypass (`RepositoryRole` id `5`) on the
   Tag ruleset (UI + `gh api`).
2. Move `<prefix>X` / `<prefix>X.Y` to the **peeled** release commit — the same
   `move_tag` semantics as `promote-release.yml`.
3. **Revert** the ruleset so moves are Release-App-exclusive again.

Notes the consumer-owned dispatch-workflow alternative. Wires step 4 of the
#1151 runbook to this subsection.

Docs-only; no code change.

Closes #1152
